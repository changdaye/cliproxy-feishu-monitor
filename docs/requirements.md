# cliproxy-feishu-monitor 需求文档

## 1. 项目背景

当前已经有一套基于 CLIProxyAPI 的 Codex 账号管理环境，能够通过管理接口查看账号状态与 Token 使用情况。

在日常使用中，核心诉求不是手动进入管理页面查看，而是：

- 定期自动获取账号可用状态
- 定期自动汇总 Token 使用量
- 将结果主动推送到飞书机器人
- 在服务异常时尽快收到提醒
- 能够以低维护成本部署到 Linux 服务器长期运行

因此需要一个轻量、稳定、可部署的监控服务，对 CLIProxyAPI 中的 Codex 账号做周期性巡检，并把结果推送到飞书。

---

## 2. 项目目标

本项目的目标是提供一个可长期运行的监控服务，用于：

1. 定期获取 CLIProxyAPI 中 Codex 账号的配额与可用状态
2. 定期统计 Token 使用量（7 小时、24 小时、7 天、累计）
3. 以中文消息格式推送到飞书机器人
4. 支持启动通知、健康心跳、异常提醒
5. 支持单次执行与常驻运行两种模式
6. 支持通过 GitHub Release tar 包直接部署到服务器

---

## 3. 目标用户

本项目面向以下用户：

- 自己维护 CLIProxyAPI 服务的人
- 需要集中监控多个 Codex OAuth 账号状态的人
- 希望通过飞书而不是手工登录后台来查看使用情况的人
- 希望低成本、轻依赖地在 Linux 服务器上长期运行监控任务的人

---

## 4. 核心使用场景

### 场景 1：日常配额巡检
用户希望每隔一段时间自动收到一条飞书消息，了解：
- 当前总账号数
- 可用账号分布（充足 / 高 / 中 / 低 / 耗尽 / 禁用 / 异常）
- 最近 7 小时、24 小时、7 天以及累计 Token 使用量

### 场景 2：服务保活确认
用户希望服务启动后立即知道服务是否正常启动，并且每隔固定时间收到健康心跳，确认监控服务仍在运行。

### 场景 3：异常告警
当 CLIProxyAPI 管理接口不可访问、查询失败或连续多次运行失败时，用户希望收到飞书告警，而不是等到很久以后才发现服务失效。

### 场景 4：服务器部署
用户希望不依赖复杂环境准备，直接通过 GitHub Release 下载 tar 包，再通过 shell 脚本一键安装成 systemd 服务。

---

## 5. 功能需求

## 5.1 数据获取
系统必须能够通过 CLIProxyAPI 管理接口获取以下信息：

1. `GET /v0/management/auth-files`
   - 用于获取 Codex 账号列表
   - 获取账号的启用/禁用状态、认证索引、账号标识等信息

2. `POST /v0/management/api-call`
   - 用于以账号身份请求 Codex 配额接口
   - 读取每个账号的 `code-5h` / `code-7d` 配额窗口

3. `GET /v0/management/usage`
   - 用于获取 Token 使用统计明细
   - 汇总 7 小时、24 小时、7 天、累计 Token 使用量

---

## 5.2 状态汇总
系统必须能够根据账号配额结果给账号打状态标签，至少包括：

- `full`：额度充足
- `high`：高可用
- `medium`：中等可用
- `low`：低可用
- `exhausted`：已耗尽
- `disabled`：账号被禁用
- `error`：查询异常
- `missing`：缺少必要信息

系统必须在每次汇总时给出：
- 总账号数
- 各状态数量
- 7 日免费等效值
- Token 用量统计

---

## 5.3 飞书推送
系统必须支持飞书机器人 webhook 推送。

推送消息要求：
- 使用中文文案
- 结构清晰，适合移动端阅读
- 包含来源地址、时间、账号状态概览、Token 汇总
- Token 用量显示为原始整数值，并带千分位

必须支持飞书签名密钥（secret）模式。

---

## 5.4 运行模式
系统必须支持两种运行模式：

### 1）run-once
- 只执行一次采集和推送
- 适合本地调试、手动巡检、部署后验证

### 2）serve
- 常驻运行
- 根据配置定时执行汇总推送与心跳推送
- 适合服务器长期运行

---

## 5.5 定时策略
当前默认策略应满足：

- 正式汇总推送：每 6 小时 1 次
- 健康心跳推送：每 3 小时 1 次
- 启动通知：开启
- 启动后立即执行一次汇总：开启

这些参数应支持通过配置文件修改。

---

## 5.6 异常处理
系统必须具备以下异常处理能力：

1. 如果单次采集失败，应记录错误信息
2. 如果连续失败次数达到阈值，应发送飞书异常告警
3. 应记录最近成功时间、最近错误时间、最近错误内容
4. 应把运行状态持久化，避免重启后丢失心跳/汇总调度状态

---

## 5.7 配置管理
系统必须支持：

- 本地 JSON 配置文件
- 环境变量回退

推荐配置文件包括：
- `local.runtime.json`
- `runtime.local.json`
- `config.local.json`

关键配置项包括：
- CPA 地址
- 管理密钥
- 飞书 webhook
- 飞书签名密钥
- 汇总间隔
- 心跳间隔
- 并发数
- 请求超时
- 失败告警阈值
- 状态文件路径

---

## 5.8 部署能力
系统必须支持：

1. 本地直接构建二进制
2. 构建 Linux x86_64 tar 包
3. 通过 `install-service.sh` 安装为 systemd 服务
4. 通过 `deploy-from-tar.sh` 从 GitHub Release tar 包一键部署
5. 提供查看服务状态、查看日志、单次执行的辅助脚本

---

## 6. 非功能需求

## 6.1 稳定性
- 服务应支持长期运行
- 服务异常退出后，systemd 应自动拉起
- 配置错误时应尽早失败并输出清晰错误信息

## 6.2 可维护性
- 代码结构清晰，职责明确
- 部署流程尽量标准化
- Release 产物应可复用
- 文档应足够覆盖首次部署场景

## 6.3 安全性
- 公开仓库中不得出现真实管理密钥、飞书密钥、运行时状态文件
- 公开文档中不得暴露真实服务器 IP
- 运行配置文件应支持单独保存在服务器本地

## 6.4 可观测性
- 每次运行应输出明确日志
- 服务启动、心跳、异常告警应可在飞书中体现
- 应能通过 `systemctl status` 和 `journalctl` 快速排查问题

---

## 7. 非目标 / 暂不支持范围

以下内容不属于当前版本目标：

- 图形化 Web 面板
- 多租户管理
- PostgreSQL / Redis 等复杂存储支持
- 复杂的权限系统
- 多消息渠道（如 Telegram、企业微信、钉钉）
- 针对 Claude / Gemini / 其他 Provider 的统一监控

当前版本聚焦：
**Codex + CLIProxyAPI + 飞书 + 服务器部署**。

---

## 8. 成功标准

项目上线后，认为满足需求的标准包括：

1. 能在服务器上稳定以 systemd 服务运行
2. 启动后能收到飞书启动通知
3. 每 6 小时能收到一条状态汇总消息
4. 每 3 小时能收到一条健康心跳消息
5. 汇总中能正确显示账号状态与 Token 使用量
6. GitHub Release tar 包可直接用于部署
7. 文档足够让用户在不看源码的前提下完成安装与使用

---

## 9. 验收标准

一个版本可被视为“可交付”时，应满足：

- `go test ./...` 通过
- 可成功构建二进制
- 可成功构建 Linux release tar 包
- `run-once --dry-run` 可正常执行
- `run-once` 可正常推送飞书
- `serve` 模式可正常启动
- Release 页面有可下载 tar 包
- README 中包含部署说明与最短部署命令

---
